def roda_com_entrada(executavel, arquivo_in):
    with open(arquivo_in) as f:
        start = time.perf_counter()
        proc = subprocess.run([executavel], input=f.read(), text=True, capture_output=True)
        end = time.perf_counter()

        # print('Sada:', proc.stdout)
        # print('Stderr:', proc.stderr)
        # print('Tempo total(s):', end - start)
        f.close()
    return (proc.stdout, end - start)

def gerador_entrada(n, m):
    for i in range(0, 10):
        f = open(f'./in-{i}.txt', 'w')
        seq=[str(n)+'\n',
            str(m)+'\n',
            ''.join(random.choices(['A','T','C','G','-'],k=n))+'\n',
            ''.join(random.choices(['A','T','C','G','-'],k=m))]
        f.writelines(seq)
    f.close()

Paralelismo com OpenMP¶

.

Uma das estratgias utilzadas para otimizar o algoritmo de sequenciamento foi utilizar a paralelizao com OpenMP. Essa biblioteca permite que o cdigo seja paralelizado por meio do lanamento de threads, fazendo o map primeiramente e o reduce no final.

No contexto do projeto, o clculo utilizado ainda era o mesmo do da Busca Exaustiva, utilizando a estratgia de Smith Waterman para encontrar o score mximo. Em termos de cdigo, as alteraes foram sucintas: apenas a adio de uma linha no loop principal.

#pragma omp parallel for firstprivate(n, m, H)

Ou seja, cada thread disponvel ir tratar do clculo do score de um par de subsequncias. Vale ressaltar que foi necessrio fazer com que a matriz de clculo H fosse um atributo firstprivate, para que todas tenham sua prpria instncia dessa varivel, garantindo o score correto.

Tambm foi necessrio criar um vetor de structs, sendo que cada struct possui trs atributos: a subsequncia A, a subsequncia B e o score. A funo de clculo devolve o score, e ele atribudo ao seu respectivo struct result, sendo que ao final percorre-se o vetor de structs para encontrar o maior score.

Para o clculo do OpenMP, foram geradas mltiplas entradas com um mesmo tamanho, de tal forma que o clculo seja rodado diversas vezes para cada n e que se obtenha uma mdia dos tempos obtidos e por consequncia adquirindo um resultado mais confivel.

    n += 5
    m += 5

Paralelismo com GPU¶

.

Uma maneira de encontrar uma soluo tima global para o problema de alinhamento de sequncias por meio da busca exaustiva, que testa todas as subsequncias possveis e encontra o par com maior alinhamento. No entanto, possui uma complexidade muito alta quando as entradas so grandes. Dessa forma, como maneira de contornar esse problema, possvel utilizar a GPU como forma de paralelismo para o clculo de score de cada subsequncia, ao invs de utilizar apenas a CPU para todo o algoritmo.

Utilizando GPU, a estratgia que originalmente se tinha em mente era de passar uma grande quantidade de dados para a GPU, ou seja, aumentar a vazo de dados e dessa forma alocar o mximo de dados possvel na GPU ao mesmo tempo. A CPU mais rpida para executar instrues, mas quando as instrues so simples ela tem um desempenho pior que o da GPU pois executa sequencialmente, enquanto que a GPU consegue paralelizar seus processos.

Por isso, o clculo de score de cada par de subsequncia foi alterado com o objetivo de obter uma maior vazo de dados. O mtodo de gerao de todas as subsequncias ainda se mantem o mesmo, de forma sequencial, bem como a forma como elas so percorridas. No entanto, o clculo no feito mais utilizando Smith Waterman. O clculo dentro da GPU realiza a comparao de uma subsequncia A com uma letra da subsequncia de B, utilizando os mesmos critrios de Smith Waterman. Dessa forma, ao invs de j montar a matriz e realizar o clculo encontrando o maior valor dela, o clculo foi alterado de tal forma a ser feito de linha e linha.

A biblioteca utilizada foi a do thrust, em que um transform com um functor utilizado para realizar o clculo descrito acima (essa comparao a da diagonal e a da insero), e um inclusive scan para a deleo. Vale ressaltar que a maneira como eram geradas mltiplas entradas para um mesmo nfoi abdicada aqui devido ao alto tempo de execuo do cdigo.

Pode-se observar que o desempenho da GPU no est bom, para entradas grandes ele demora muito tempo para executar. Uma possvel iterao ao projeto poderia ser criar um grande vetor do tipo char que possuiria todas as subsequncias de uma sequncia 'concatenadas' e outro vetor de ndices que faria o controle de faixas para obter reconstruir as subsequncias corretas. Dessa forma, seria possvel enviar todos os dados de uma vez para a GPU, o que tornaria o desempenho do cdigo muito melhor. No entanto, no foi possvel realizar a construo desse algoritmo, pois a biblioteca thrust no permite que funes do tipo host (CPU) sejam utilizadas dentro de functors, impossibilitando o clculo (Uma tentativa de fazer o cdigo dessa ideia est no arquivo tentativa_GPUindex.cu, dentro da pasta Paralelismo, porm o cdigo no foi finalizado).
